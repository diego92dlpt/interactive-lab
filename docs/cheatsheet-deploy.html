Deploying to Production

From a working local build to live at interactive-lab.vercel.app.

How to read this: Each step shows the action in bold — that's all you need to glance at. Click "▶ Details / Why" only if you need more context or a reminder of why something matters.

Phase 1 — Pre-Flight Check

1
Run npm run build — must complete with zero errors
Details / Why

This runs the same build process Vercel will run. If it fails locally, it will fail on Vercel too — better to catch it now.

A successful build ends with ✓ built in Xs and lists the output files. Any errors or warnings in red need to be fixed first.

Run this from the wait-calc-sim\ folder in a terminal.

Phase 2 — Commit and Push to GitHub

2
Check what's changed: git status
Details / Why

This shows you all modified and new files. Glance through it — make sure nothing unexpected appears (like .env files or temp folders you didn't intend to commit).

3
Stage your changes — be specific about which files
# Stage specific files (preferred): git add src/ index.html package.json # Or stage everything modified (fine if you reviewed step 2): git add -A
Details / Why

Staging means "include these in the next commit". git add src/ stages all changes inside the src/ folder.

The dist/ folder (the built output) is in .gitignore and will be ignored automatically — Vercel rebuilds from source.

If you only changed experiment files: git add src/experiments/your-name/ is precise and safe.

4
Commit with a clear message
git commit -m "Add epidemic simulation experiment" # or git commit -m "Fix: [describe what was broken]"
Details / Why

The commit message is for your future self. Keep it short but descriptive — "Add X" for new features, "Fix X" for bug fixes.

If git asks for your name/email the first time, run:
git config --global user.email "you@example.com"
git config --global user.name "Your Name"

5
Push to GitHub
git push origin main
Details / Why

This sends your commits to GitHub (github.com/diego92dlpt/interactive-lab). Vercel watches this repo and automatically triggers a deployment the moment a push to main lands.

If git asks for credentials, use your GitHub username and a Personal Access Token (not your password) — GitHub disabled password auth in 2021.

Phase 3 — Watch the Vercel Deployment

6
Go to vercel.com → your interactive-lab project → watch the deployment status
Details / Why

After your push, a new deployment appears at the top of the list with status Building. It typically takes 1–2 minutes.

It will go: Queued → Building → Ready (green checkmark). If it turns red with Error, click it to see the build log — the error message is usually clear.

7
Once status is Ready, open the production URL — not a preview URL
https://interactive-lab.vercel.app
Details / Why — what's the difference between all those URLs?

Vercel shows several URLs in the deployment panel. Here's what they are:

interactive-lab.vercel.app — your production URL. This is what's public. Always use this one to verify.

interactive-lab-abc123xyz.vercel.app (with a hash) — a preview URL unique to this specific deployment. Useful for sharing a specific version, but not the canonical public address.

interactive-lab-git-main-xyz.vercel.app — a branch-specific preview. Ignore for production testing.

Rule of thumb: if the URL has a hash or "git" in it, it's a preview. Use the clean URL.

8
Test in a fresh incognito / private window
Details / Why

Your browser may have cached an old version of the site. An incognito window starts fresh with no cache, so you see exactly what a new visitor would see.

Test: home page loads → new experiment card appears → clicking it routes correctly → experiment works end-to-end.

Phase 4 — If Something Is Broken

9
Diagnose: is the bug present locally too?

Decision Tree

Bug is local too Fix the code → npm run build → commit → push. Vercel redeploys automatically.
Works locally, broken on Vercel Likely a build-time issue. Run npm run build locally and look carefully at warnings/errors. Common causes: missing file, wrong import path casing (Windows is case-insensitive, Linux is not), an env variable that exists locally but not on Vercel.
Vercel build failed (red) Click the failed deployment → "View Build Logs". The error is usually on the last few lines. Fix it locally, confirm npm run build passes, then push again.
Details / Why — the casing issue

The most common "works locally, fails on Vercel" issue: import paths with wrong casing. Windows doesn't care if you write import X from "./Components/foo" vs ./components/foo" — both work. Linux (which Vercel runs on) treats them as different files and will fail.

Always match import paths exactly to actual folder/file names.

10
After fixing: commit and push again — Vercel redeploys automatically
git add src/ git commit -m "Fix: [describe the fix]" git push origin main
