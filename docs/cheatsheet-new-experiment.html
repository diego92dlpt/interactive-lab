Interactive Lab — Cheatsheet 1 of 2

Adding a New Experiment

From idea to locally-running prototype, ready for deployment.

How to read this: Each step shows the action in bold — that's all you need to glance at. Click "▶ Details / Why" only if you need more context or a reminder of why something matters.

Phase 1 — Set Up the Experiment Folder

1
Create the experiment folder
src/experiments/your-experiment-name/
Details / Why

Use kebab-case (lowercase, hyphens, no spaces). This slug will become part of the public URL — e.g. /experiments/epidemic-sim.

Inside this folder, each experiment is self-contained: its own index.jsx, a components/ subfolder, and any logic files.

2
Drop your documentation into a docs/ subfolder inside it
src/experiments/your-experiment-name/docs/ requirements.txt ← or .md, .pdf — anything works design-notes.txt ...
Details / Why

Claude Code can read .txt, .md, .pdf, and most text formats. Keeping docs here means you can start a fresh session and simply say "read everything in this folder and give me a plan" — no copy-pasting needed.

This also keeps experiment-specific context out of the project root.

Phase 2 — Wire It Into the App (do this early)

3
Register the route in src/App.jsx
// Add this import at the top: import MyExperiment from "./experiments/your-experiment-name"; // Add this Route inside <Routes>: <Route path="/experiments/your-experiment-name" element={<MyExperiment />} />
Details / Why

The path string here is what users see in the URL bar. It should match the folder name you chose in Step 1.

Do this early — even before Claude Code starts building — so you can navigate to the experiment in the browser while it's being built.

4
Add the experiment card in src/pages/Home.jsx
// Add an entry to the experiments array: { title: "My Experiment", description: "One sentence describing what this does.", tag: "Category Label", link: "/experiments/your-experiment-name", },
Details / Why

The tag appears as the small green monospace label above the title (e.g., "Physics Simulation", "Epidemiology"). Pick something short and descriptive.

The link must exactly match the path you registered in App.jsx.

Wire this up early too — the card on the home page will work immediately and clicking it will route to your experiment as it's being built.

Phase 3 — Build with Claude Code

5
Start a new Claude Code session. First prompt: ask it to read and plan — don't ask it to build yet
"Please read all files in src/experiments/<name>/docs/ and give me a summary of what I'm building and a phased implementation plan."
Details / Why

Letting CC read and summarize first surfaces any misunderstandings early, before code is written. Review its plan — make sure it has the right mental model before saying "go".

Ask for a phased plan if the experiment is complex. Phases let you test each piece before building the next.

6
Approve the plan, then build one phase at a time
Details / Why

Don't ask CC to build everything in one shot on complex experiments. A phased approach means each phase is testable and any mistakes are caught early instead of tangled into a large codebase.

Example prompts: "Build phase 1 only" → test → "Phase 1 looks good, proceed to phase 2".

7
Test constantly at http://localhost:5173 — keep npm run dev running in the background
Details / Why

Open a terminal in the project folder and run npm run dev. Leave it running the whole session — it hot-reloads automatically on every file save, so you see changes instantly without restarting.

To open a terminal in the right folder: navigate to wait-calc-sim\ in File Explorer, right-click → "Open in Terminal".

Phase 4 — Wrap Up Locally

8
Run npm run build — it must complete with zero errors before deploying
Details / Why

npm run dev is forgiving — it lets some things slide. npm run build is strict and runs the same process Vercel uses. If it fails locally, it will fail on Vercel too.

Common things it catches: unused imports that are actually missing, syntax errors, broken asset paths.

Run this in the terminal from the project folder. A successful build ends with ✓ built in Xs.
