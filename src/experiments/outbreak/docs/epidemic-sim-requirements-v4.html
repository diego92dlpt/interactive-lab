01 Tool Overview & Naming locked

This tool is Experiment #2 on the Interactive Lab portal (interactive-lab.vercel.app). It consists of two distinct simulation modes housed in a single tool, accessed from a shared landing page where the user chooses which mode to enter.

ModeNameLanding page description
ABM Outbreak Watch a pathogen move through a population in real time. Configure transmission, immunity, and quarantine behavior — then let 1,000 individuals find their own fate.
Monte Carlo Scenario Lab Build a detailed epidemic model and run it hundreds of times. Tune demographics, disease parameters, policies, and behaviors — then compare outcomes across configurations to understand what actually moves the needle.
Scenario Lab's full scope is documented in sections 10–17 of this document.
02 Report Export & Config Persistence Scenario Lab only

Scenario Lab supports two persistence features: report export for sharing and record-keeping, and config save/load for returning to a previous parameter set. The original file-based compare tool has been superseded by the in-session what-if comparison system documented in Section 10.

Report export

After a run completes, the user can export the full report as a self-contained HTML file. This file is for sharing and archiving — not for comparison (comparison is in-session). The exported file contains:

Exported HTML structure

Visible report — scenario name, description, date/time of run, curated key parameter summary, all charts and tables from the Output & Analytics section.

Invisible JSON block — full config snapshot embedded as <script type="application/json" id="scenario-config">. Never rendered in the browser. Used only for config reload.

Config save and load

The user can export their current config as a JSON file and reload it in a future session. On load, all sliders and toggles repopulate to the saved values. This allows returning to a previous parameter set without rebuilding from scratch.

Config load is also triggered by loading a previously exported HTML report — the tool reads the embedded JSON block and restores all parameters. The user can then modify settings and run a new scenario.

Scenario name and description

User-provided before running. Name is required — Run button is gated until a name is entered. Description is optional free text for context. Both appear prominently in the exported report header.

No session memory

No auto-save. No backend. No persistent state between browser sessions. If the user closes the tab without exporting, their run is lost. This is by design — the tool is stateless.

What was superseded

The original design included a file-based twin-pane compare tool with synchronized scrolling and format validation. This has been replaced entirely by the in-session what-if comparison system (Section 10). Two scenarios are now compared live within a single session using the same random seed — no file management required.

03 Disease Model — Outbreak locked

State machine

Six mutually exclusive states. Each dot is always in exactly one state. Transitions are one-way only — no backward transitions, no reinfection.

StateLabelMeaningTransitions to
SSusceptibleHealthy, no immunity, can be infectedE
EExposedInfected, incubating, not yet infectiousI
IInfectiousActively spreading diseaseR or D
RRecoveredNo longer infectious; partial or full immunity—
DDeadFrozen in place, × marker, stays on canvas—
VVaccinatedPre-immune at sim start; partial or full immunity—

Transmission mechanics

Every S×I dot collision triggers an independent Bernoulli trial (coin-toss) at probability p. Success = S transitions to E. No distributions, no memory, no correlation between events.

R×I collisions use the same coin-toss with effective probability = p × (1 − RI%). At RI = 100% the collision is inert. At RI = 0% recovered dots are as vulnerable as susceptible.

V×I collisions use the same coin-toss with effective probability = p × (1 − VI%). At VI = 100% the collision is inert. VI% and RI% are independent sliders — vaccine-derived and infection-derived immunity are not assumed to be equal.

E×any collisions do not trigger transmission. E-dots are infected but not yet infectious — they participate in Brownian motion and physical bouncing normally, but never act as a transmission source.

Mask wearing — per-collision modifier

Applies to all transmission-eligible collisions: S×I, R×I, and V×I. The table below uses S-dot as shorthand for the receiving dot, but the same formula applies when the receiving dot is R or V (with their respective immunity factors applied first, then the mask modifier).

I-dot maskedS-dot maskedEffective transmission probability
NoNop
YesNop × (1 − ME%)
NoYesp × (1 − ME%)
YesYesp × (1 − ME%)²

Mask assignment is per-dot at initialization, coin-tossed at probability MW%. Never changes during run.

Age/demographic variation in p or IFR · asymptomatic carriers as a distinct state · immunity waning · reinfection · stochastic parameter variation (all params are point estimates).

05 Contact & Mobility Model — Outbreak locked

The arena

Single rectangular box, fixed 16:9 aspect ratio (landscape). Hard wall boundaries — dots bounce off walls. No zones, no geography. Responsive to device orientation: on portrait tablets, aspect ratio rotates/refits to use available space. Dot radius is auto-tuned relative to box size and N — not user-exposed.

Brownian motion

Each dot has position (x, y) and a velocity vector (direction + speed). Each tick: velocity applied to position, then a small random directional nudge (JITTER_FACTOR) perturbs direction to produce organic wandering. Two named constants live at top of config/constants file:

BASE_SPEED — canonical starting velocity magnitude before temperature scaling

JITTER_FACTOR — magnitude of per-tick random directional nudge

Temperature

Three discrete settings: L / M / H. Controls average dot speed via a scalar applied to BASE_SPEED. Auto-normalized for density: BASE_SPEED = k × (box_area / N)^0.5 where k_L, k_M, k_H are tunable constants. This ensures LMH feels consistently meaningful regardless of population size. Temperature affects all dots except compliant I-dots (stationary) and D-dots (frozen).

Collision detection — two types

Dot-dot: collision = distance between centers ≤ sum of radii. S×I pairs trigger transmission coin-toss. R×I pairs trigger partial immunity coin-toss. Physical elastic bounce on all collisions (velocity exchange). A named boolean constant DOT_DOT_ELASTIC toggles this off to pass-through if performance testing requires it.

Dot-fence: collision = distance from dot center to fence center ≤ fence radius. Per-DOT-FENCE pair flag prevents re-evaluation mid-penetration. Flag released when dot distance > fence radius (checked every tick). Fence is circular.

Quarantine fence mechanics (detailed)

When does a fence appear?
When an I-dot is coin-tossed as compliant (per QC%), it stops moving and a static circular fence F is instantiated at 3× dot radius, centered on that I-dot.
Overlapping fences?
Each fence is an independent entity. A dot can carry simultaneous pass-through flags for multiple fences with no conflict.
Fence lifecycle?
Destroyed immediately when its I-dot transitions to R or D.
Non-compliant I-dots?
Continue moving freely at normal temperature. No fence is created. Behave identically to pre-quarantine model.
Fence visual treatment?
Low-opacity dashed ring. Subtle — present but not visually dominant. Does not obscure dots moving nearby.
All other dots vs fence?
100% bounce off fence (compliance model is on I-dots, not on other dots respecting the fence). S×I collision detection inside the fence continues unchanged.

10 Scenario Lab — Overview locked

Scenario Lab is the Monte Carlo simulation mode. No live animation. The user configures a detailed disease model, optionally sets up a what-if comparison scenario, then runs 100–1000 simulations. Results are presented as statistical distributions, not single outcomes. The core pedagogical goal is illuminating the flaw of averages — showing that average inputs produce a distribution of outcomes, not a single average outcome.

Pedagogical focus

Three specific MCS lessons are woven into the tool's design:

Lesson 1 — Stochastic extinction
At R₀ = 2.5, roughly 30% of outbreaks fizzle naturally from a single introduction. The average predicts explosive growth. The distribution reveals a bimodal world of fizzles and explosions.
Lesson 2 — Superspreader overdispersion
Two diseases with identical R₀ but different k produce completely different outbreak distributions. Same average, wildly different risk profiles. A formula cannot show this.
Lesson 3 — Compounding variance
Pre-symptomatic window, asymptomatic fraction, and infectious duration each have variance. That variance compounds across transmission chains. The distribution of outcomes is far wider than any formula predicts.

What-if comparison system

Before running, the user may optionally configure a what-if scenario alongside the baseline. The flow:

Step 1
User configures baseline — all sliders, toggles, and parameters set as desired. Does not hit Run yet.
Step 2 (optional)
A checkbox/button asks: "Compare with a what-if scenario?" Default = No. If Yes, a modal opens pre-populated with all baseline settings. User changes 1–3 parameters to create the what-if variant.
Step 3
User hits Run. Both scenarios execute using the same random seed, same MC run count, same N. Results appear in twin-pane side-by-side view.
Locked vs free parameters between baseline and what-if

Locked (cannot differ): random seed · N · MC run count · max sim-days · initial infected · V% · M%

Free to differ: any disease model parameter · any intervention/policy parameter · contact rate c

Design philosophy

Modern and polished visual design — distinct from Outbreak's retro aesthetic. Not generic AI-generated UI. Full details deferred to Section 17.

Distribution implementation

All duration parameters use a shifted negative binomial: draw = floor + NegBin(mean − floor, k). Floor = 1 for both PSD and SD. User always sets and sees real-world mean. The gamma distribution is the theoretically correct continuous fit; NegBin is the standard discrete approximation for day-resolution simulations, with a slightly heavier right tail that conservatively overstates tail risk.

R₀ dispersion k uses a standard NegBin with floor = 0 — infecting zero people is a valid and common real-world outcome.

13 Scenario Lab — Contact & Mobility Model locked

Homogeneous random mixing. No zones, no geography, no Brownian motion. Every person has a fixed daily contact rate — the average number of other people encountered per sim-day, drawn randomly from the full population.

ParameterRangeDefaultNotes
c — daily contact rate1–5010Point estimate. No distribution. Tooltip: urban 8–15, rural 2–5.

c is deliberately exposed as a user-facing slider (not derived from R₀) to enable urban vs rural comparisons with all other parameters held equal. R₀ is the emergent readout: R₀ ≈ p × c × SD_mean. Users set p and c independently; R₀ falls out.

Only S, V, and M state people can receive new infection events. E, P, I, A, D cannot be infected. Each transmission-eligible contact between an infectious person and a susceptible person triggers the appropriate coin toss.

Quarantine mechanics

Only I-state people can quarantine — P and A never quarantine because they don't know they're sick. This structural ceiling is a key teaching point: symptom-based quarantine cannot intercept pre-symptomatic or asymptomatic transmission regardless of compliance rate.

Compliant I-people reduce daily contacts to c_q = 1 (fixed internal constant, not user-exposed) for the remainder of their infectious duration.

Contact tracing formula

CT% applies to P and A state people — tracing finds pre-symptomatic and asymptomatic contacts of confirmed cases regardless of symptoms, via PCR testing.

Since no contact history is stored (homogeneous random mixing has no memory), tracing reach is computed probabilistically each sim-day using the coupon collector formula:

CT% reach formula

P(person reached by tracing) = 1 − (1 − 1/N)^(I × c × L)

Where L = 7-day lookback window (fixed internal constant). For each P and A person each day: coin toss at CT% × P(reached). If isolated: contacts reduce to c_q = 1 for remainder of their state duration.

Naturally saturates at 1.0 as I grows — never produces absurd values exceeding population size. At small I: tracing reach is low and roughly linear. At large I: reach saturates toward CT%, making further case growth the binding constraint.

Key teaching point: CT% = 90% sounds impressive. But in high-k overdispersed outbreaks, most transmission happens before any case is confirmed. Tracing is always chasing the past. MCS reveals the distribution of "tracing success" — bimodal in overdispersed diseases, working well in slow runs, nearly useless in explosive ones.

The report is structured into nine buckets. Buckets 1–7 are visual and interactive. Bucket 8 is parked for future consideration. Bucket 9 is raw data export. All buckets appear in a single scrolling report page with a sticky mini-navigation bar for jumping between sections.

Design philosophy — entire tool

One consistent modern and polished design language across the entire tool — landing page, Outbreak module, and Scenario Lab module. No retro elements anywhere. The reference aesthetic is tools like Linear, Vercel's dashboard, or Observable: clean typographic hierarchy, generous whitespace, purposeful use of color, no decorative chrome. The kind of UI that signals craft rather than generation.

PrincipleDetail
Color modeDark default · light mode toggle available · consistent across both modules

Touch/tabletNative throughout — no mouse assumptions in either module · adequate tap targets on all interactive elements

Emergent R₀ readout

Persistent live display in the Disease — transmission section. Updates whenever p, c, or SD mean changes. Formula: R₀ ≈ p × c × SD_mean. Labeled "Observed R₀ (emergent)" with a small note: "Determined by p and c — not a direct input." Styled prominently — larger than surrounding parameter values, clearly a derived result not a setting.

NegBin distribution widget

An inline mini-histogram appears below each parameter that uses a NegBin distribution: PSD mean/k, SD mean/k, and R₀ k. Updates live as mean or k changes. Shows the current distribution shape at a glance — the user immediately sees how changing k widens or sharpens the tail.

Every such parameter also shows a "learn more →" link. All instances open the same shared NegBin explainer modal regardless of which parameter triggered it. The modal is general — it teaches NegBin as a concept, not any one parameter specifically.

NegBin explainer modal contents

Why gamma is the theoretically correct continuous fit for duration data · Why NegBin is the appropriate discrete approximation for day-resolution simulation · What k means visually — interactive histogram showing shape at different k values · The floor/shifted distribution explanation (draw = floor + NegBin(mean − floor, k)) · Why lower k means more variance, not less, despite the zero bar growing taller · The COVID-19 and Spanish Flu k values as concrete anchors

Preset templates

"Load preset" button near the top of the config panel, above the parameter sections. Opens a modal showing five historical templates. Selecting a preset populates all disease model parameters (sections 3–5). Intervention and simulation settings are unchanged. Confirmation dialog: "This will overwrite your current disease settings. Intervention settings will not be changed." After loading, the emergent R₀ readout should closely match the known historical value — a built-in sanity check that the preset is well-calibrated.

What-if modal

Triggered by the what-if checkbox at the bottom of the config panel. Opens a full-screen modal (large drawer on tablet).

What-if modal structure

Header: "What-if scenario — adjust 1–3 parameters to compare against your baseline."

Body: Full config panel replicated, pre-populated with all baseline values. Same section structure, same slider/type-in/tooltip pattern.

Changed parameter highlighting: Any parameter that differs from baseline gets a subtle colored left border or background tint. Immediately visible which parameters have been modified.

Differences summary panel: Persistent sidebar (or bottom panel on mobile) listing only changed parameters — "p: 15% → 25%" — gives a clean audit of what differs between the two scenarios.

Locked parameters: N · random seed · MC runs · max sim-days · initial infected · V% · M% — shown but grayed out and uneditable. Tooltip explains why: "Must match baseline for valid paired comparison."

Actions: Cancel (discards changes, returns to config) · Confirm (saves what-if config, returns to config with what-if indicator active).
